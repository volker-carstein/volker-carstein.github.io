Aparté Test d'intrusion (pentest)

  • Un test d'intrusion est une évaluation de sécurité mandatée : un professionnel simule une attaque réelle dans un cadre contractuel précis, avec pour objectif d'identifier des vulnérabilités exploitables.
  • La portée est définie à l'avance par un document juridique (lettre de mission / Rules of Engagement) : périmètre technique (IPs, domaines, applications), types d'attaques autorisées, fenêtres horaires. Rien ne se fait hors périmètre.
  • L'objectif livrable est un rapport : liste des vulnérabilités, preuves d'exploitation (screenshots, outputs), niveau de criticité (CVSS ou autre), recommandations de remédiation.

Point de vue client

  • Résultat concret et actionnable
  • Durée et coût maîtrisés
  • Permet de corriger des failles précises

Point de vue opérateur

  • Couvrir le maximum de surface dans le temps imparti
  • Documenter rigoureusement chaque trouvaille
  • Respecter scrupuleusement le périmètre

Aparté Red Team

  • Une mission de red team simule une attaque réaliste et persistante inspirée d'un scénario adversarial crédible (APT, compromission fournisseur, insider threat, etc.). L'objectif n'est pas de faire l'inventaire de toutes les vulnérabilités, mais de tester la capacité de l'organisation à détecter et répondre à une intrusion réelle.
  • La red team opère souvent à l'insu des équipes défensives (blue team) — c'est la condition pour mesurer leur réactivité réelle. Elle opère sur une durée longue (semaines à mois) avec un périmètre très large, voire full-scope.
  • Les TTPs employées cherchent à imiter un/des acteur(s) réel(s) : évasion de détection, persistance, OPSEC rigoureux — l'objectif est de ne pas se faire détecter tout en atteignant les cibles définies.

Point de vue client

  • Mesure la maturité sécurité globale (humains, processus, outils)
  • Révèle les lacunes de détection, pas seulement les failles techniques

Point de vue opérateur

  • Priorité à la discrétion et à la persistance
  • Rapport centré sur le chemin d'intrusion complet et les "detection gaps"

Active Directory — Architecture de base

  • Active Directory Domain Services (AD DS) est le service d'annuaire de Microsoft, présent dans la quasi-totalité des entreprises de taille significative. Il centralise l'authentification, les autorisations et la gestion de configuration du parc Windows.
  • Structure hiérarchique : Forêt > Domaine(s) > Unités d'Organisation (OU) > Objets. La forêt est la vraie frontière de sécurité — pas le domaine.
  • Le Contrôleur de Domaine (DC) héberge la base AD (NTDS.dit), gère les authentifications (Kerberos, NTLM) et répond aux requêtes LDAP. Tous les DCs d'un domaine se répliquent entre eux — compromettre l'un d'eux suffit à compromettre le domaine entier.
  • Les objets AD sont accessibles en lecture LDAP à tout utilisateur authentifié du domaine — conception volontaire de Microsoft pour la résolution de noms et les applications métier. Cela signifie qu'un attaquant avec un seul compte peut cartographier l'intégralité de l'annuaire.

Types d'objets et de comptes dans AD

Objets principaux

  • User — compte d'utilisateur humain
  • Computer — chaque machine joinée au domaine possède un compte MACHINE$ (MDP 120 car., rotation 30j)
  • Group — Security group (permissions) ou Distribution group (messagerie)
  • OU — Unité d'Organisation : conteneur pour délégation et application de GPO

Comptes de service

  • Service accounts — comptes dédiés à l'exécution d'applications, souvent avec SPN et mot de passe permanent
  • gMSA — mot de passe géré et renouvelé automatiquement par AD (bonne pratique)

Comptes built-in importants

  • Administrator (RID 500) — admin local de la machine, ou admin du domaine sur les DCs
  • krbtgt (RID 502) — compte de service du KDC, désactivé, son hash chiffre tous les TGTs du domaine
  • Guest (RID 501) — compte invité, désactivé par défaut

Groupes privilégiés clés

  • Domain Admins — admin complet sur le domaine
  • Enterprise Admins — admin sur toute la forêt
  • Domain Controllers — les DCs eux-mêmes
  • Backup Operators, Server Operators — droits sensibles sur les DCs

GPO — Group Policy Objects

  • Les GPO appliquent des configurations centralisées à des users et machines : sécurité, scripts, logiciels, pare-feu, politique de mots de passe. Elles sont liées à un Site, un Domaine ou une OU (ordre LSDOU — les plus proches l'emportent).
  • Stockées dans SYSVOL (\\corp.local\SYSVOL\), accessible en lecture à tout utilisateur du domaine.

Intérêt défensif

  • Désactiver LLMNR/NBT-NS
  • Activer SMB signing, LAPS
  • Politique de mots de passe fine (PSO)
  • Restriction des droits admin locaux

Intérêt offensif

  • SYSVOL peut contenir des credentials (anciens scripts GPP)
  • Modifier une GPO liée à un DC → exécution de code sur tous les DCs
  • Identifier les configurations faibles déployées en masse

SID — Security Identifier

  • Le SID est l'identifiant unique et permanent d'un principal de sécurité (user, groupe, ordinateur). Contrairement au nom, il ne change pas lors d'un renommage — c'est lui que Windows utilise en interne pour toutes les décisions d'autorisation.
  • Format : S-1-5-21-[domaine1]-[domaine2]-[domaine3]-[RID] — les trois valeurs identifient le domaine, le RID (Relative Identifier) identifie l'objet dans ce domaine.
  • Le PAC (Privilege Attribute Certificate), extension Microsoft intégrée aux tickets Kerberos, contient la liste des SID de l'utilisateur (son propre SID + SID de tous ses groupes). Les services lisent le PAC pour décider des autorisations sans re-contacter le DC.

Authentification NTLM — Mécanisme

Challenge-response à 3 messages — aucun mot de passe ni hash transmis en clair

1. NEGOTIATE Client ──────→ Serveur : annonce les capacités supportées

2. CHALLENGE Serveur ─────→ Client : nonce aléatoire de 8 octets (challenge)

3. AUTHENTICATE Client ──────→ Serveur

└─ Net-NTLMv2 = HMAC-MD5(NT_hash, challenge + client_nonce + timestamp + ...)

  • Le NT hash n'est jamais transmis — seul le résultat du HMAC-MD5 (Net-NTLMv2) voyage sur le réseau. Le serveur vérifie en recalculant de son côté avec le hash stocké, ou via Netlogon passthrough vers le DC.
  • Net-NTLMv2 ≠ NT hash : on ne peut pas utiliser un Net-NTLMv2 capturé pour du Pass-the-Hash — mais on peut le cracker offline ou le relayer en temps réel vers une autre cible.
  • NTLM est utilisé automatiquement en fallback : accès par IP, service sans SPN, clients hors domaine — il est pratiquement impossible de l'éliminer complètement.

NT Hash — Un credential Windows intéressant

  • Windows ne stocke jamais les mots de passe en clair — il stocke uniquement leur empreinte NT hash : NT_hash = MD4(mot_de_passe_UTF-16LE). Hash non-salé de 16 octets (128 bits).
  • Absence de sel = vulnérabilité structurelle : deux comptes avec le même mot de passe ont exactement le même NT hash — attaque en parallèle possible, rainbow tables applicables.
  • Le NT hash est stocké dans NTDS.dit (sur les DCs, l'équivalent de /etc/shadow pour tout le domaine) et dans la base SAM locale (comptes locaux de chaque machine).
  • En mémoire, LSASS (Local Security Authority Subsystem Service) maintient les NT hashes des sessions actives. Les extraire nécessite les droits SYSTEM ou SeDebugPrivilege — c'est une des actions a effectuer en priorité après la compromission d'une machine.
Le NT hash est un credential à part entière — il suffit à s'authentifier via NTLM sans connaître le mot de passe en clair

Les hashes comme credentials à part entière

  • L'une des propriétés les plus importantes — et les plus dangereuses — du protocole NTLM est que le NT hash lui-même est le credential : il n'est pas un dérivé intermédiaire du vrai mot de passe, c'est directement ce qui est utilisé dans le protocole d'authentification.
  • Conséquence directe : si on obtient le NT hash d'un compte, on peut s'authentifier en tant que ce compte sur n'importe quel service NTLM, sans jamais connaître le mot de passe en clair. C'est la technique du Pass-the-Hash.
  • Les NT hashes se trouvent à deux endroits principaux : en mémoire dans le processus LSASS (pour les sessions actives) et dans le fichier NTDS.dit sur les DCs (pour tous les comptes du domaine). Accéder à l'un ou l'autre avec les droits appropriés suffit à extraire tous les hashes.
  • Pour Kerberos, le NT hash sert de clé symétrique pour chiffrer la pré-authentification — ce qui permet l'Overpass-the-Hash : utiliser le NT hash pour obtenir un TGT Kerberos valide, même sans connaître le mot de passe en clair.

Kerberos V5 — Rappel du flux

Protocole d'authentification par défaut dans AD — basé sur des tickets à durée de vie limitée

1. AS-REQ Client ──→ KDC (AS) : identité (clair) + pré-auth [timestamp chiffré avec le secret du client]

2. AS-REP KDC ────→ Client

├─ TGT [chiffré avec hash krbtgt — opaque pour le client] contient : identité, PAC, clé session Kc-tgs, validité

└─ Partie client [chiffrée avec le secret du client] contient : clé session Kc-tgs

3. TGS-REQ Client ──→ KDC (TGS) : TGT + authenticator [chiffré avec Kc-tgs] + SPN souhaité

4. TGS-REP KDC ────→ Client

├─ Service Ticket (ST) [chiffré avec hash du compte de service — opaque] contient : PAC, identité, clé session Kc-srv

└─ Partie client [chiffrée avec Kc-tgs] contient : clé session Kc-srv

5. AP-REQ Client ──→ Service : ST + authenticator [chiffré avec Kc-srv — prouve la possession de Kc-srv]

6. AP-REP Service ─→ Client : [optionnel] timestamp+1 chiffré avec Kc-srv — authentification mutuelle

Étapes 5–6 : le service déchiffre le ST avec son propre hash et valide l'authenticator — le KDC n'est plus consulté

Kerberos V5 — Le PAC et les points d'attaque

  • Le PAC (Privilege Attribute Certificate) est signé avec le hash krbtgt (signature KDC) et le hash du compte de service (signature serveur). Forger un PAC valide nécessite donc le hash krbtgt (Golden Ticket) ou celui du compte de service (Silver Ticket).
  • C'est le PAC que les services lisent pour les autorisations — pas les groupes AD en temps réel. Si on forge le PAC, on contrôle les droits que le service nous accorde.

ACL / DACL / ACE — Le modèle de permissions AD

  • Chaque objet AD possède un Security Descriptor qui contient sa DACL (Discretionary Access Control List). La DACL est une liste d'ACE (Access Control Entry) — chaque ACE définit ce qu'un principal peut (ou ne peut pas) faire sur l'objet.
  • Une ACE = un triplet : (principal, type de droit, allow/deny). Exemple : "alice — Full Control — Allow" sur le compte bob signifie qu'alice peut tout faire sur bob.
  • Les ACE sont héritées des OUs parentes. Une ACE placée sur une OU s'applique automatiquement à tous les objets enfants — une seule permission mal configurée peut compromettre des centaines d'objets.
  • Les mauvaises configurations d'ACE s'accumulent sur des années sans audit. Elles constituent souvent le chemin d'attaque le plus direct vers des privilèges élevés — sans exploiter aucune CVE.

Le compte krbtgt — Cible (plus que) privilégiée

  • Le compte krbtgt est le compte de service du KDC. Son NT hash est la clé symétrique utilisée pour chiffrer et signer tous les TGT émis dans le domaine — c'est le secret le plus critique d'un domaine AD.
  • Obtenir le NT hash de krbtgt permet de forger n'importe quel TGT (Golden Ticket) : n'importe quel utilisateur, n'importe quels droits, n'importe quelle durée de validité — sans contact avec le DC.
  • Le compte krbtgt est désactivé par défaut et doit le rester. Son mot de passe ne change que sur action manuelle explicite de l'administrateur.
  • Pour invalider un Golden Ticket déjà émis, il faut changer le mot de passe de krbtgt deux fois à au moins 10 minutes d'intervalle (AD maintient les deux derniers hashes comme valides pour éviter les coupures de service).

Active Directory comme surface d'attaque

  • La centralisation d'AD — identités, authentifications, autorisations, configurations — en fait la cible prioritaire. Compromettre AD, c'est compromettre l'ensemble du SI.
  • Les ransomwares modernes (Conti, LockBit...) exploitent systématiquement AD pour se propager latéralement et chiffrer le réseau en quelques heures après le foothold initial.
  • Le schéma d'attaque est quasi-constant : compte bas-privilège → énumération LDAP → exploitation de mauvaises configurations → escalade vers Domain Admin. Aucune CVE n'est nécessaire si les configurations sont faibles.

Cibles prioritaires

  • NTDS.dit sur les DCs — hashes de tous les comptes
  • LSASS en mémoire — hashes et tickets actifs
  • krbtgt — clé de tout le domaine

Vecteurs typiques

  • Mauvaises configurations ACE
  • Comptes de service avec mot de passe faibles
  • Délégation Kerberos mal configurée
  • Absence de SMB Signing

Partie 1

Reconnaissance

Énumération & Cartographie de l'environnement

Pourquoi la reconnaissance est-elle primordiale ?

  • La phase de reconnaissance est souvent celle qui détermine le succès ou l'échec d'une attaque. Un attaquant qui comprend parfaitement son environnement cible choisit le chemin de moindre résistance — celui qui fait le moins de bruit et qui atteint le plus rapidement l'objectif.
  • Active Directory est conçu pour être interrogeable par tout utilisateur authentifié : LDAP est accessible en lecture à n'importe quel compte du domaine par défaut. Un attaquant avec un seul compte bas-privilège peut cartographier l'intégralité de l'annuaire sans déclencher la moindre alarme.
  • Cette accessibilité totale en lecture est une décision de conception de Microsoft — AD doit être utilisable, et les applications ont besoin de résoudre des noms, des groupes, des attributs. La conséquence est qu'aucune information dans l'annuaire ne peut être considérée comme secrète du point de vue d'un attaquant authentifié.
  • Les outils modernes comme BloodHound automatisent entièrement cette phase : en quelques minutes, une cartographie complète du domaine est disponible, avec les chemins d'attaque calculés automatiquement.

LDAP — La mine d'information accessible à tous

  • Le protocole LDAP (port 389 / 636 en TLS) est le protocole d'interrogation d'AD. Toute requête LDAP avec un filtre approprié retourne les objets correspondants et leurs attributs — y compris des attributs très sensibles comme servicePrincipalName, userAccountControl, description, memberOf.
  • Une requête LDAP typique pour trouver tous les utilisateurs du domaine : (&(objectCategory=person)(objectClass=user)). Ajouter (userAccountControl:1.2.840.113556.1.4.803:=4194304) filtre sur le flag DONT_REQ_PREAUTH.
  • Les attributs retournés par défaut incluent déjà tout ce dont un attaquant a besoin pour cibler ses premières attaques. Des outils comme ldapdomaindump aspirent l'intégralité de l'annuaire en une commande.
Un audit de sécurité commence toujours par une phase d'énumération LDAP — si votre DC ne répond pas à des requêtes LDAP filtrées, c'est un problème de configuration, pas une protection

Ce qu'on cherche — Comptes vulnérables

  • Comptes avec DONT_REQ_PREAUTH : ces comptes peuvent être attaqués sans aucun credential préalable via AS-REP Roasting — ce sont des cibles prioritaires pour obtenir un premier accès.
  • Comptes avec SPN (Service Principal Names) : tout compte utilisateur (non-machine) avec un SPN est attaquable par Kerberoasting — on peut obtenir un hash crackable offline avec n'importe quel compte authentifié du domaine.
  • Comptes avec adminCount=1 : ces comptes sont ou ont été membres de groupes privilégiés à un moment. Leur DACL est contrôlée par AdminSDHolder et ils constituent des cibles de haute valeur.
  • Comptes avec mot de passe n'ayant jamais expiré ou comptes inactifs : les comptes de service sont souvent configurés avec des mot de passe permanents qui n'ont pas changé depuis des années — ce qui les rend vulnérables au Kerberoasting avec des wordlists ciblées.
  • Champ description non vide : il est remarquablement fréquent en entreprise de trouver des mots de passe ou des informations sensibles dans la description d'un compte — erreur d'administration classique.

Ce qu'on cherche — Infrastructure et permissions

  • Machines avec Unconstrained Delegation (flag TRUSTED_FOR_DELEGATION, hors DCs) : ces machines stockent les TGTs des utilisateurs qui s'y connectent — les compromettre permet de voler des tickets de haute valeur.
  • Relations d'approbation (trusts) : une confiance entre domaines peut permettre à un attaquant ayant compromis un domaine de se propager dans un domaine adjacent, surtout si la confiance est bidirectionnelle.
  • ACE intéressantes sur des objets sensibles : GenericAll sur un compte DA, WriteDACL sur le domaine, AddMember sur Domain Admins... Ces permissions accordées par erreur constituent souvent le chemin d'attaque le plus direct.
  • Membres des groupes privilégiés : connaître exactement qui est Domain Admin, Backup Operator, etc. permet de cibler les comptes qui donnent accès aux ressources les plus sensibles avec le moins d'étapes possibles.
  • Sessions actives (HasSession) : savoir qu'un admin est actuellement connecté sur une machine permet de cibler cette machine pour in-fine voler son TGT et Pass-the-Ticket.

SPN — Service Principal Names en détail

  • Un SPN est un identifiant unique d'une instance de service dans AD. Il associe un service à son compte de domaine — c'est ce qui permet à Kerberos de savoir quel hash utiliser pour chiffrer le Service Ticket correspondant.
  • Format général : serviceClass/hostname[:port[/instanceName]]. Exemples courants :
MSSQLSvc/srv-sql01.corp.local:1433   → SQL Server
HTTP/intranet.corp.local             → IIS / Application web
CIFS/fileserver.corp.local           → Partage de fichiers
LDAP/DC01.corp.local                 → LDAP sur DC (toujours présent)
  • Chaque SPN est enregistré sur un objet AD (compte user ou machine). Si le SPN est sur un compte utilisateur, le ST correspondant sera chiffré avec le NT hash de ce compte — exploitable par Kerberoasting.
  • Les comptes machine ont des SPNs aussi, mais leurs mot de passe font 120 caractères aléatoires et se renouvellent tous les 30 jours — pratiquement impossible à cracker, donc pas intéressants pour Kerberoasting.

Attaques sur les Credentials

Password Spraying · AS-REP Roasting · Kerberoasting

Vue d'ensemble — Obtenir un premier accès

  • Cette partie couvre les attaques qui permettent d'obtenir un premier credential valide sur le domaine, ou d'élever les privilèges à partir d'un compte bas-privilège en récupérant le mot de passe d'un compte plus intéressant.
  • Ces attaques ont en commun d'exploiter des faiblesses de configuration dans AD ou des mots de passe faibles — elles ne nécessitent pas de vulnérabilité logicielle et fonctionnent sur tout AD, même entièrement patché.
  • Elles sont particulièrement puissantes car elles ne génèrent aucune alerte par défaut sur un AD standard — AS-REP Roasting et Kerberoasting ressemblent à des requêtes d'authentification légitimes.

Password Spraying — Principe et timing

  • Le Password Spraying consiste à tester un mot de passe unique contre un grand nombre de comptes, contrairement au brute force classique qui teste de nombreux mots de passe sur un seul compte. Cette inversion permet d'éviter le verrouillage de compte.
  • La plupart des politiques AD verrouillent un compte après 3 à 10 tentatives échouées dans une fenêtre de temps (souvent 30 minutes). En ne testant qu'un seul mot de passe par fenêtre temporelle, on reste en dessous du seuil indéfiniment.
  • L'efficacité est surprenante : dans un domaine de 500 utilisateurs, si seulement 1% ont un mot de passe faible (ce qui est statistiquement quasi-certain), cela représente 5 comptes compromis avec une seule vague de spray.
  • Il faut au préalable connaître la politique de verrouillage du domaine — requête LDAP sur l'attribut lockoutThreshold de l'objet domaine. L'outil kerbrute (via Kerberos, donc sans logs LDAP) est particulièrement discret.
Le Password Spraying est souvent la toute première attaque tentée après énumération des comptes — elle ne laisse quasiment aucune trace si bien exécutée

Password Spraying — Construction des wordlists

  • Mots de passe saisonniers : Printemps2024!, Automne2024, Noel2024! — ce pattern est extrêmement répandu car les utilisateurs doivent régulièrement changer leur mot de passe et choisissent souvent la saison + l'année courante.
  • Mots de passe dérivés du nom de l'organisation : NomEntreprise2024, NomEntreprise!, NomEntreprise123 — souvent utilisés comme mot de passe initial ou par les utilisateurs qui veulent "s'en souvenir facilement".
  • Mots de passe universellement faibles : Welcome1, P@ssword1, Password123!, Azerty1! — ces mot de passe satisfont les critères de complexité Windows (majuscule, minuscule, chiffre, caractère spécial) tout en étant triviaux à deviner.
  • Mots de passe de bienvenue standards : de nombreux services informatiques utilisent un mot de passe standard pour les nouveaux comptes et espèrent que l'utilisateur le changera — ce qu'il ne fait souvent pas.
kerbrute passwordspray -d corp.local users.txt "Automne2024!"
CrackMapExec smb dc01.corp.local -u users.txt -p "Automne2024!" --no-bruteforce

AS-REP Roasting — La pré-authentification Kerberos

  • Par défaut, lorsqu'un client demande un TGT (AS-REQ), il doit inclure un authentifiant de pré-authentification : un timestamp chiffré avec son NT hash. Le KDC déchiffre cet authentifiant pour vérifier que le client connaît bien son mot de passe avant de lui répondre.
  • Ce mécanisme est crucial : sans pré-authentification, le KDC répondrait à n'importe qui demandant un TGT pour n'importe quel compte — et la réponse AS-REP contiendrait une partie chiffrée avec le hash de ce compte, permettant une attaque offline.
  • L'attribut DONT_REQ_PREAUTH du flag userAccountControl désactive cette protection pour un compte spécifique. Certains vieux clients Kerberos (ou certaines applications mal configurées) ne supportent pas la pré-auth, ce qui conduit les admins à désactiver cette protection.
  • Cette configuration est une mauvaise pratique grave qui expose directement le hash du compte à une attaque offline, sans qu'un attaquant n'ait besoin du moindre credential préalable.

AS-REP Roasting — Mécanisme d'attaque

// alice a DONT_REQ_PREAUTH activé dans son userAccountControl

1. Attaquant ──→ KDC : AS-REQ pour alice (aucune pré-auth fournie)

2. KDC ────────→ Attaquant : AS-REP

├─ TGT chiffré avec hash krbtgt [opaque, inutile]

└─ Clé de session [chiffrée avec NT hash d'alice] ← ON CRAQUE ÇA

3. Attaquant → hashcat -m 18200 → mot de passe d'alice en clair

  • L'attaque peut être menée sans aucun credential si null session LDAP est possible (possible dans de rares configs) ou avec n'importe quel compte du domaine pour l'énumération.
  • Le hash retourné est au format $krb5asrep$23$ — crackable avec hashcat mode 18200 et une bonne wordlist (rockyou, règles custom).
  • BloodHound identifie directement les comptes concernés via la propriété dontreqpreauth: true sur les nœuds User.

AS-REP Roasting — Outils et commandes

# Impacket — avec un compte du domaine pour énumérer automatiquement

# Rubeus — depuis Windows
Rubeus.exe asreproast /format:hashcat /outfile:asrep.txt

# Crack offline avec hashcat

  • Si le mot de passe cracké donne accès à un compte de service ou à un compte avec des ACE intéressantes, c'est souvent le point d'entrée qui ouvre toute la chaîne d'escalade.

Kerberoasting — Pourquoi les comptes de service sont vulnérables

  • Tout utilisateur authentifié du domaine peut demander au KDC un Service Ticket (ST) pour n'importe quel SPN enregistré dans AD. C'est une fonctionnalité normale de Kerberos — un client demande un ticket pour se connecter à un service.
  • Le Service Ticket est chiffré avec le NT hash du compte propriétaire du SPN. Si ce compte est un compte utilisateur (pas machine), ce NT hash est dérivé d'un mot de passe défini par un humain — et donc potentiellement faible et crackable.
  • Les comptes de service AD créés manuellement (pour des applications, des services Windows, des tâches planifiées) ont souvent des mots de passe : définis au départ sans politique stricte, jamais changés depuis des années, parfois documentés dans des fichiers de configuration, et souvent partagés entre plusieurs comptes.
  • Cette attaque est fondamentalement différente de l'AS-REP Roasting car elle ne nécessite aucune mauvaise configuration — elle exploite le fonctionnement normal de Kerberos. Tout compte de service avec un SPN user est vulnérable par design.

Kerberoasting — Mécanisme d'attaque

// svc-sql a le SPN : MSSQLSvc/srv-sql01.corp.local:1433

1. Attaquant ──→ KDC : TGS-REQ avec TGT valide + SPN de svc-sql

2. KDC ────────→ Attaquant : TGS-REP

└─ ST [chiffré avec NT hash de svc-sql] ← ON CRAQUE ÇA

3. Attaquant → hashcat -m 13100 (RC4) ou -m 19700 (AES256)

4. Mot de passe de svc-sql récupéré → authentification directe

  • L'attaquant demande des STs pour tous les comptes avec SPN en une seule passe — aucune interaction avec les services cibles n'est nécessaire.
  • Le KDC génère les tickets sans suspicion car la demande est parfaitement légitime — n'importe quel client est censé pouvoir demander un ST pour n'importe quel service.
  • Les STs récupérés sont crackés offline, sans aucune interaction supplémentaire avec le réseau — technique totalement passive après la collecte.

Kerberoasting — RC4 vs AES et ciblage

  • RC4-HMAC (type 23) : le chiffrement basé directement sur le NT hash. C'est le type historique, encore très répandu. Crackage avec hashcat mode 13100 — vitesse de plusieurs milliards de tentatives/seconde sur GPU moderne.
  • AES256-CTS-HMAC-SHA1 (type 18) : dérive une clé depuis le mot de passe via PBKDF2. Environ 10× plus lent à cracker que RC4. Mais un attaquant peut forcer la demande en RC4 en spécifiant etype=23 dans la TGS-REQ, même si le compte supporte AES.
  • Kerberoasting ciblé : via une ACE GenericWrite (ou sa variante minimale WriteSPN) sur un compte utilisateur, on peut ajouter un SPN à n'importe quel compte (même un DA) pour le rendre instantanément Kerberoastable. BloodHound CE matérialise la permission minimale comme arête WriteSPN — contrairement à GenericWrite, elle ne donne accès qu'à servicePrincipalName, pas à msDS-KeyCredentialLink.

# Hashcat — RC4
hashcat -m 13100 kerb.txt wordlist.txt -r rules/best64.rule

# Rubeus — forcer RC4
Rubeus.exe kerberoast /tgtdeleg /rc4opsec /outfile:kerb.txt

Pass-the-Hash (PtH)

  • Le Pass-the-Hash exploite le fait que NTLM utilise directement le NT hash dans le protocole challenge-response. Au lieu de dériver le NT hash depuis un mot de passe, on l'injecte directement dans les structures d'authentification Windows.
  • Une fois un NT hash extrait (depuis LSASS ou NTDS.dit), on peut ouvrir des connexions SMB, WinRM, RPC ou HTTP authentifiées en tant que le compte propriétaire du hash — sans jamais saisir le mot de passe.
  • Cette technique fonctionne jusqu'à ce que le hash extrait soit invalide (rotation de mot de passe). Elle fonctionne aussi pour le compte Administrator local si le RID est 500 (pas les admins locaux non-RID500 depuis KB2871997).
# Mimikatz — ouvrir un shell en tant que alice avec son hash

Overpass-the-Hash

  • L'Overpass-the-Hash consiste à utiliser un NT hash pour initier une authentification Kerberos. Le NT hash est utilisé comme clé pour chiffrer l'authentifiant de pré-authentification (timestamp) — exactement comme le ferait le client Kerberos légitime.
  • Le résultat est un TGT Kerberos valide pour le compte, obtenu sans connaître le mot de passe en clair. Ce TGT peut ensuite être utilisé pour demander des Service Tickets pour n'importe quel service du domaine.
  • Particulièrement utile dans les environnements où NTLM est restreint ou monitoré mais où Kerberos reste disponible. Permet aussi de convertir un hash récupéré offline en session Kerberos active.
# Mimikatz — obtenir un processus avec TGT Kerberos injecté
sekurlsa::pth /user:alice /domain:corp.local /ntlm:aabbcc... /run:powershell.exe

# Rubeus — demander le TGT et l'injecter directement
Rubeus.exe asktgt /user:alice /rc4:aabbcc... /ptt

# Rubeus — demander le TGT et l'écrire dans un fichier
Rubeus.exe asktgt /user:alice /rc4:aabbcc... /outfile:alice.kirbi

Pass-the-Ticket (PtT)

  • Les tickets Kerberos (TGT et STs) sont stockés en mémoire LSASS. Ils peuvent être exportés en fichiers .kirbi et importés sur une autre machine — sans connaître le hash ni le mot de passe. Le ticket reste valide jusqu'à son expiration (10h par défaut pour un TGT).
  • Scénario typique : un admin est connecté sur la machine compromise → son TGT est en mémoire → export → import sur la machine attaquante → session Kerberos complète avec ses droits, sans bruit.
# Mimikatz — exporter tous les tickets
sekurlsa::tickets /export

# Rubeus — lister et exporter
Rubeus.exe dump /service:krbtgt /nowrap

# Importer un ticket (inject dans la session courante)
Rubeus.exe ptt /ticket:alice.kirbi
kerberos::ptt alice.kirbi  # Mimikatz

Silver Ticket — Forger un Service Ticket

  • Le Silver Ticket consiste à forger complètement un Service Ticket en utilisant le NT hash du compte de service. On ne demande rien au KDC — on crée le ticket de toutes pièces localement, en y inscrivant le PAC de notre choix (nom, SID, groupes).
  • Rappel : le Service Ticket est validé par le service lui-même avec son propre hash (étapes 5/6 de Kerberos) — le DC n'intervient pas dans cette validation. Un ticket correctement forgé sera donc accepté sans que le DC ne soit jamais contacté, sans trace dans les logs du DC.
  • On choisit dans le PAC forgé un utilisateur Administrator avec les SID des groupes Domain Admins — le service croira qu'il communique avec un administrateur de domaine, alors que le compte n'existe peut-être même pas.
# Mimikatz — forger un Silver Ticket pour CIFS/srv-fileserver
kerberos::golden /user:Administrator /domain:corp.local /sid:S-1-5-21-... \
  /target:srv-fileserver.corp.local /service:cifs \
  /rc4:aabbcc... /ptt   # rc4 = NT hash du compte machine srv-fileserver$
Aucune trace sur le DC — le ticket ne transite jamais par le KDC après sa création

Golden Ticket — Forger un TGT

  • Forger un TGT complet avec le NT hash de krbtgt — le ticket est accepté par tout KDC. On inscrit dans le PAC les SID voulus (Domain Admins, Enterprise Admins...), pour un compte qui peut ne pas exister.
  • Persistance remarquable : le ticket reste valide même si le mot de passe d'Administrator est changé ou si le compte "forgé" est supprimé. Il n'est invalidé que par une double rotation du mot de passe krbtgt.
  • La double rotation de krbtgt est l'action prioritaire post-compromission DC — mais elle doit être planifiée avec soin (risque de casser des sessions en cours).
# Mimikatz — créer un Golden Ticket
kerberos::golden /user:FakeAdmin /domain:corp.local /sid:S-1-5-21-... \
  /krbtgt:aabbcc... /groups:512,519,544 /ptt

Diamond Ticket — La variante discrète

  • Le Diamond Ticket est une variante moderne du Golden Ticket qui cherche à être moins détectable. Au lieu de forger un TGT de zéro, on demande un vrai TGT au KDC pour un compte légitime, puis on déchiffre le PAC avec le hash krbtgt, on le modifie (en ajoutant des SID de groupes privilégiés), et on rechiffre.
  • L'avantage est que le TGT a un horodatage et une structure parfaitement légitimes — les solutions de détection basées sur l'analyse des tickets (anomalies de timestamp, RID inexistants...) seront moins susceptibles de le détecter qu'un Golden Ticket classique.
  • Cette technique est plus récente et requiert les mêmes prérequis que le Golden Ticket (hash krbtgt), mais peut passer à travers certaines détections qui bloquent les Golden Tickets. C'est une illustration de la course permanente entre techniques d'attaque et de détection.
# Rubeus — Diamond Ticket
Rubeus.exe diamond /krbkey:<aes256_krbtgt> /user:alice /password:alice_password \
  /enctype:aes /ticketuser:Administrator /ticketuserdomainsid:S-1-5-21-... \
  /groups:512 /ptt

Partie 4

NTLM & Attaques par Relai

Poisoning · SMB Relay · LDAP Relay · Coercition

Pourquoi NTLM reste un vecteur d'attaque majeur

  • Malgré Kerberos comme protocole par défaut, NTLM est utilisé automatiquement dans de nombreux cas : accès à une ressource par adresse IP plutôt que par nom DNS, service sans SPN enregistré, fallback en cas d'échec Kerberos, clients hors domaine, et beaucoup d'applications legacy qui forcent explicitement NTLM.
  • Désactiver NTLM complètement est un objectif louable mais extrêmement difficile en pratique : il casse régulièrement des applications, des scripts d'administration, des anciennes imprimantes réseau, et des services tiers qui n'ont pas été mis à jour.
  • NTLM est donc présent dans la quasi-totalité des réseaux Windows, et il présente une vulnérabilité fondamentale : les challenges NTLM peuvent être relayés. Contrairement à Kerberos, NTLM n'intègre pas de protection contre le relai (pas de liaison au canal de transport).
  • Les attaques NTLM Relay sont d'autant plus puissantes qu'elles peuvent être entièrement automatisées et combinées avec d'autres techniques (coercition d'authentification, LDAP relay pour RBCD) pour construire des chaînes d'attaque complexes.

LLMNR / NBT-NS / mDNS — Protocoles de résolution de noms

  • LLMNR (Link-Local Multicast Name Resolution, RFC 4795) et NBT-NS (NetBIOS Name Service) sont des protocoles de résolution de noms en broadcast/multicast qui entrent en jeu quand la résolution DNS échoue. Ils sont activés par défaut sur tous les Windows.
  • Scénario typique : un utilisateur fait une faute de frappe en accédant à \\FILSERVEUR au lieu de \\FILESERVER. DNS ne connaît pas ce nom → Windows envoie une requête LLMNR en broadcast sur le LAN : "Quelqu'un connaît FILSERVEUR ?".
  • N'importe quelle machine sur le LAN peut répondre "Oui, c'est moi" — Windows accepte la première réponse reçue sans vérification d'authenticité. La victime envoie alors automatiquement ses credentials NTLM à l'attaquant pour s'authentifier.
  • mDNS (Multicast DNS, utilisé notamment par les Macs et Linux) est également sensible au même type de poisoning dans certains contextes.
Ces protocoles sont inutiles dans un AD bien configuré avec DNS fonctionnel — leur désactivation est une recommandation de durcissement standard (GPO)

Responder — Capturer les hashes Net-NTLMv2

  • Responder est l'outil de référence pour l'exploitation de LLMNR/NBT-NS/mDNS. Il écoute sur toutes les interfaces réseau, répond à toutes les requêtes de résolution de noms, et présente de faux serveurs SMB, HTTP, FTP, SQL... pour capturer les authentifications.
  • Les hashes Net-NTLMv2 capturés peuvent être crackés offline avec hashcat (mode 5600). Un GPU récent teste plusieurs centaines de millions de candidats par seconde : avec rockyou.txt, un mot de passe basé sur un mot courant tombe en quelques secondes ; un mot de passe plus élaboré peut résister des heures à des jours.
  • Attention : Responder répond aussi aux requêtes SMB et HTTP par défaut. Quand on prépare un NTLM Relay, il faut désactiver SMB et HTTP dans Responder (Responder.conf) pour que l'outil ntlmrelayx puisse capter les connexions et les relayer sans conflits de port.
# Lancer Responder en mode capture
python3 Responder.py -I eth0 -rdwv

# Cracker les hashes capturés
hashcat -m 5600 ntlmv2.txt /usr/share/wordlists/rockyou.txt
Les hashes capturés sont sauvegardés dans /usr/share/responder/logs/

NTLM Relay — Principe du man-in-the-middle

Plutôt que de cracker le hash, on le réutilise en temps réel en se positionnant entre la victime et une cible tierce. La clé : transmettre à la victime le même nonce reçu de la cible — sa réponse NTLMv2 est alors valide pour la cible.

1. Victime ───→ Attaquant : NEGOTIATE

2. Attaquant ──→ Cible : NEGOTIATE

3. Cible ──────→ Attaquant : CHALLENGE (nonce C)

4. Attaquant ──→ Victime : CHALLENGE (même nonce C)

5. Victime ───→ Attaquant : AUTHENTICATE (NTLMv2 sur nonce C)

6. Attaquant ──→ Cible : AUTHENTICATE (même réponse)

→ Attaquant est authentifié en tant que Victime sur la Cible

Condition nécessaire : la signature SMB/LDAP doit être désactivée sur la cible — sinon le relay échoue car les paquets sont signés avec une clé que l'attaquant ne possède pas

SMB Signing — La protection principale contre le relay

  • Le SMB Signing (signature des paquets SMB) empêche le relay SMB en signant cryptographiquement chaque paquet avec une clé dérivée de la session. Sans accès à cette clé, l'attaquant ne peut pas créer des paquets SMB valides à relayer.
  • Par défaut : SMB Signing est requis sur les DCs mais seulement supporté (non requis) sur les workstations et serveurs. C'est pourquoi le SMB relay vise généralement les workstations et non les DCs.
  • Le LDAP Signing est souvent plus permissif : il faut le LDAP Channel Binding (EPA) pour bloquer le LDAP Relay — et ce paramètre n'est pas activé par défaut sur toutes les versions de Windows Server.

Bloque le relay ✓

  • SMB Signing required
  • LDAP Signing required
  • LDAP Channel Binding (EPA)

Vulnérable ✗

  • SMB Signing disabled/optional (workstations)
  • LDAP sans Channel Binding
  • HTTP sans EPA

SMB Relay — Exploitation

  • Le SMB Relay cible des machines sans SMB Signing obligatoire. Si la victime est administrateur local sur la cible, on obtient une exécution de commande ou un shell avec les droits SYSTEM.

LDAP Relay — Possibilités offensives

  • Relayer une authentification NTLM vers LDAP sur le DC permet d'effectuer des opérations LDAP en tant que la victime. Si la victime a des droits suffisants dans AD, les possibilités sont très larges.
  • Créer un compte machine : par défaut, tout utilisateur du domaine peut créer jusqu'à 10 comptes machine (MachineAccountQuota = 10). Ce compte contrôlé sera utilisé ensuite pour configurer RBCD (cf. Partie 6).
  • Modifier des ACE : si la victime a WriteDACL sur le domaine ou un objet sensible, on peut s'attribuer des droits DCSync ou GenericAll.
  • Configurer RBCD : si la victime a GenericWrite sur un Computer, on peut modifier msDS-AllowedToActOnBehalfOfOtherIdentity pour préparer une attaque RBCD (cf. Partie 6).

  --no-smb-server --no-http-server

Coercition d'authentification — Forcer une cible à s'authentifier

  • Les attaques de relay et PtT sont encore plus puissantes quand on peut forcer une machine cible à s'authentifier vers nous — c'est la coercition d'authentification.
  • PrinterBug / SpoolSample : exploite le protocole MS-RPRN (Print Spooler). On demande à une machine distante de notifier un changement d'imprimante vers notre machine → la machine cible s'authentifie vers nous en NTLM.
  • PetitPotam : exploite MS-EFSRPC (Encrypting File System Remote Protocol). Force une machine à s'authentifier vers nous. Particulièrement utile pour forcer un DC à s'authentifier.
  • DFSCoerce, ShadowCoerce : variantes utilisant d'autres protocoles RPC (DFS, VSS) pour la même finalité. Ces techniques évoluent constamment à mesure que des protocoles sont patchés.
La combinaison coercition d'authentification + NTLM Relay est une chaîne fréquemment exploitée en red team pour progresser dans un AD sans credentials initiaux

Partie 5

Abus des ACLs & DACLs

Permissions mal configurées → escalade de privilèges

Le modèle de permissions AD — Rappel complet

  • Dans AD, chaque objet possède un Security Descriptor qui contient quatre éléments : Owner (propriétaire), DACL (Discretionary ACL), SACL (System ACL, pour l'audit), et Group. La DACL liste les ACE qui définissent qui peut faire quoi sur l'objet.
  • Les ACE peuvent être héritées depuis les OUs parentes ou définies directement. Une ACE héritée d'une OU s'applique à tous les objets descendants — une erreur de permission à ce niveau peut compromettre des centaines d'objets en une seule ACE.
  • Les objets membres de groupes privilégiés (Domain Admins, etc.) ont leur DACL remplacée toutes les 60 minutes par la DACL du container AdminSDHolder via le processus SDProp. Cela signifie que ces comptes sont "protégés" mais aussi que modifier AdminSDHolder permet une persistance durable.
  • L'audit des ACE en entreprise est une tâche complexe car elles s'accumulent sur des années. BloodHound est le seul outil pratique pour analyser l'ensemble des permissions d'un AD à grande échelle.

GenericAll sur un User — Exploitation

  • GenericAll sur un compte utilisateur donne un contrôle total sur cet objet. Les vecteurs d'exploitation les plus discrets sont ceux qui ne modifient pas le mot de passe directement (car un changement de mot de passe est loggé et l'utilisateur peut remarquer qu'il ne peut plus se connecter).
  • Targeted Kerberoasting : ajouter un SPN au compte cible (servicePrincipalName) puis demander un ST — le hash obtenu est celui du compte cible. On retire le SPN ensuite pour ne pas laisser de trace.
  • Shadow Credentials : modifier l'attribut msDS-KeyCredentialLink du compte pour y ajouter une clé publique contrôlée. On peut ensuite s'authentifier avec la clé privée via PKINIT — technique récente qui ne change pas le mot de passe et génère peu de logs.
  • ForceChangePassword : changer brutalement le mot de passe — efficace mais visible (l'utilisateur ou les monitors le remarquent).
# PowerView — Targeted Kerberoasting via GenericAll
Set-DomainObject -Identity alice -Set @{serviceprincipalname="fake/spn.corp.local"}
# Kerberoaster alice, puis nettoyer
Set-DomainObject -Identity alice -Clear serviceprincipalname

GenericAll / AddMember sur un Groupe

  • Avoir GenericAll ou AddMember sur un groupe privilégié est l'un des chemins d'escalade les plus directs. On s'ajoute au groupe et on hérite immédiatement de tous ses droits.
  • Attention : l'ajout à un groupe privilégié génère un Event ID 4728 (membre ajouté à un groupe de sécurité global) dans les logs du DC — c'est un événement surveillé dans tout SOC sérieux. Préférer des techniques plus silencieuses quand possible.
  • Alternative plus discrète : si on a GenericWrite sur une machine, configurer RBCD (cf. Partie 6) pour obtenir un accès admin local sans toucher aux groupes de domaine.
# PowerView — s'ajouter à Domain Admins
Add-DomainGroupMember -Identity "Domain Admins" -Members "attacker"

# Net — équivalent natif Windows
net group "Domain Admins" attacker /add /domain

# Vérifier (pour confirmer)
Get-DomainGroupMember -Identity "Domain Admins"

WRITE_DAC / WRITE_OWNER — La chaîne de contrôle

  • WRITE_OWNER ("Modifier le propriétaire" dans ADUC, WriteOwner dans BloodHound) — prendre la propriété d'un objet donne implicitement WRITE_DAC dessus, même sans ACE explicite.
  • WRITE_DAC ("Modifier les autorisations" dans ADUC, WriteDACL dans BloodHound) — réécrire la DACL de l'objet : on peut s'y attribuer GenericAll ou n'importe quel droit.
  • Chaîne classique : WRITE_OWNER → propriété → WRITE_DAC implicite → GenericAll → contrôle total.
  • Cas critique : WRITE_DAC sur l'objet Domaine → s'attribuer DS-Replication-Get-Changes-All → DCSync sur tout le domaine.
# S'attribuer DCSync rights via WRITE_DAC sur le domaine (PowerView)
Add-DomainObjectAcl -TargetIdentity "DC=corp,DC=local" `
  -PrincipalIdentity attacker -Rights DCSync

# DCSync
lsadump::dcsync /user:krbtgt /domain:corp.local

AdminSDHolder — Mécanisme et exploitation

  • L'AdminSDHolder est un container spécial (CN=AdminSDHolder,CN=System,DC=corp,DC=local) dont la DACL sert de modèle de sécurité pour tous les comptes privilégiés. Le processus SDProp s'exécute toutes les 60 minutes et copie cette DACL vers tous les membres actuels et anciens des groupes protégés.
  • Objectif légitime : assurer que les comptes privilégiés ne peuvent pas être modifiés par des administrateurs délégués qui auraient des droits sur les OUs parentes. C'est une protection, pas une faille.
  • Vecteur d'attaque : si on obtient WriteDACL sur AdminSDHolder, on peut y ajouter une ACE donnant GenericAll à notre compte. À la prochaine exécution de SDProp (≤ 60 min), notre ACE se propage sur tous les comptes protégés du domaine.
  • Vecteur de persistance durable : même si un admin supprime notre ACE directement sur un compte cible, SDProp la remet automatiquement toutes les 60 minutes. La seule façon de stopper ça est de supprimer l'ACE d'AdminSDHolder lui-même.

DCSync — Répliquer le domaine sans toucher au DC

  • Au lieu d'accéder physiquement à NTDS.dit, on imite un DC secondaire demandant une réplication partielle via MS-DRSR. Le DC répond sans vérifier la légitimité de la demande — il suffit d'avoir les droits adéquats.
  • Droits requis sur l'objet Domaine : DS-Replication-Get-Changes + DS-Replication-Get-Changes-All. Normalement réservés à Domain Admins, Enterprise Admins et DCs — mais toute ACE mal configurée peut les déléguer (cf. WRITE_DAC sur l'objet Domaine).
  • Détection : Event ID 4662 sur le DC avec le GUID {1131f6aa-9c07-11d1-f79f-00c04fc2dcd2} depuis une IP qui n'est pas un DC — c'est l'alerte DCSync standard dans un SIEM.
# Mimikatz
lsadump::dcsync /user:krbtgt /domain:corp.local

# Impacket (depuis Linux)
secretsdump.py corp.local/attacker:password@dc01.corp.local -just-dc-ntlm

Délégation Kerberos

Unconstrained · Constrained · RBCD

Pourquoi la délégation Kerberos existe

  • Problème fondamental de l'architecture à plusieurs niveaux : un utilisateur s'authentifie sur un serveur web (niveau 1), et ce serveur web doit accéder à une base de données SQL (niveau 2) en utilisant l'identité de l'utilisateur — et non sa propre identité. C'est le problème du double hop Kerberos.
  • Sans délégation, le serveur web ne peut s'authentifier auprès du serveur SQL que sous sa propre identité (le compte service IIS), pas celle de l'utilisateur. La délégation Kerberos résout ce problème en permettant au serveur web d'obtenir un ticket Kerberos valide pour l'utilisateur et de l'utiliser pour se connecter au SQL.
  • La délégation existe en trois variantes de sécurité croissante : Unconstrained (la plus ancienne et la plus dangereuse), Constrained (plus contrôlée), et Resource-Based Constrained (RBCD, la plus moderne et la plus sécurisée mais aussi la plus exploitable via les ACE).
La délégation est une feature légitime et nécessaire dans de nombreuses architectures — ce n'est pas une vulnérabilité en soi, mais ses mauvaises configurations le sont

S4U2Self et S4U2Proxy — Extensions Kerberos

  • S4U2Self (Service for User to Self) : permet à un service d'obtenir un Service Ticket pour un utilisateur arbitraire, adressé à lui-même, sans que cet utilisateur n'ait initié de demande. Le service prouve qu'il agit de façon légitime pour cet utilisateur. Cela crée un "proxy ticket" qui peut ensuite être utilisé avec S4U2Proxy.
  • S4U2Proxy (Service for User to Proxy) : permet à un service d'utiliser un ST obtenu via S4U2Self (ou fourni par l'utilisateur) pour demander un nouveau ST vers un autre service, toujours au nom de l'utilisateur original. C'est ce mécanisme qui fait le "double hop".
  • Offensivement, S4U2Self est précieux car il permet d'obtenir un ST pour Administrator (ou n'importe quel utilisateur) sans que celui-ci se soit jamais authentifié — à condition d'avoir compromis un compte avec les bons flags de délégation ou d'avoir configuré RBCD.

Unconstrained Delegation — Mécanisme

  • Flag : TRUSTED_FOR_DELEGATION sur le compte machine ou de service. Quand un utilisateur s'authentifie auprès d'un service avec ce flag activé, Kerberos inclut automatiquement le TGT de l'utilisateur dans l'AP-REQ (étape 5), et ce TGT est stocké en mémoire par le service.
  • L'idée originale était que le service en délégation non contrainte puisse se connecter à n'importe quel autre service en utilisant ce TGT stocké — d'où "non contrainte". En pratique, c'est un risque de sécurité majeur car la machine stocke les TGTs de tous les utilisateurs qui s'y connectent.
  • Les contrôleurs de domaine ont ce flag activé par défaut (normal — ils sont KDC). C'est sur les serveurs non-DC que cette configuration est dangereuse et souvent inutile. Les serveurs IIS, les serveurs de fichiers qui ont ce flag activé "parce que quelqu'un en avait besoin un jour" sont des cibles prioritaires.
Dans BloodHound CE, ces machines sont identifiables par leur propriété unconstraineddelegation = true — c'est l'un des premiers filtres à appliquer lors d'une investigation

Unconstrained Delegation — Attaque avec PrinterBug

  • L'attaque combine deux éléments : une machine avec Unconstrained Delegation que l'on contrôle (SRV01), et la coercition d'authentification d'un DC via le Printer Bug (SpoolSample). Le DC forcé à s'authentifier vers SRV01 verra son TGT stocké dans la mémoire de SRV01.

// SRV01 : Unconstrained Delegation activée — compromis par l'attaquant

1. Attaquant sur SRV01 : Rubeus.exe monitor /interval:5 /nowrap

2. SpoolSample.exe DC01.corp.local SRV01.corp.local

→ Force DC01$ à s'authentifier vers SRV01 via MS-RPRN

3. Rubeus capte le TGT de DC01$ → base64

4. Rubeus.exe ptt /ticket:<base64-TGT-DC01$>

5. secretsdump.py -k -no-pass DC01.corp.local

→ Tous les hashes du domaine via DCSync avec le TGT de DC01$

DC01$ (le compte machine du DC) dispose des droits de réplication → son TGT suffit pour un DCSync complet

Constrained Delegation — Mécanisme

  • La délégation contrainte a été introduite pour limiter les risques de l'Unconstrained Delegation. L'attribut msDS-AllowedToDelegateTo liste explicitement les SPNs vers lesquels le compte peut déléguer — il ne peut pas "sauter" vers d'autres services.
  • Le compte utilise S4U2Self pour obtenir un ST pour l'utilisateur cible, puis S4U2Proxy pour utiliser ce ST afin d'accéder au service listé dans msDS-AllowedToDelegateTo. Le KDC vérifie que la délégation est autorisée avant de délivrer le ticket final.
  • Avantage sur Unconstrained : le TGT de l'utilisateur n'est pas stocké — seul un ST limité au service autorisé est généré. Si le service est compromis, l'attaquant ne peut déléguer que vers les services listés.

// svc-web : msDS-AllowedToDelegateTo = MSSQLSvc/srv-sql01.corp.local

svc-web → KDC : S4U2Self pour "Administrator" (→ ST adressé à svc-web)

svc-web → KDC : S4U2Proxy avec ST → MSSQLSvc/srv-sql01

→ ST pour MSSQLSvc en tant qu'Administrator

Constrained Delegation — Protocol Transition et attaque

  • Le flag TRUSTED_TO_AUTH_FOR_DELEGATION active le Protocol Transition (aka S4U2Self étendu) : le service peut utiliser S4U2Self pour n'importe quel utilisateur sans avoir leur TGT. Sans ce flag, S4U2Self nécessite que l'utilisateur soit préalablement authentifié auprès du service.
  • Avec Protocol Transition activé, compromettre le compte de service suffit pour usurper l'identité de n'importe quel utilisateur du domaine vers les services listés — y compris impersonner Administrator pour accéder à SQL, CIFS, LDAP...
# Rubeus — S4U complet depuis le hash du compte de service
Rubeus.exe s4u /user:svc-web /rc4:<hash_svc-web> \
  /impersonateuser:administrator \
  /msdsspn:"MSSQLSvc/srv-sql01.corp.local:1433" \
  /altservice:cifs /ptt

# Impacket
getST.py -spn "MSSQLSvc/srv-sql01.corp.local:1433" \
  -impersonate Administrator \
  corp.local/svc-web -hashes :<hash>
L'option /altservice permet de modifier le SPN dans le ticket forgé — utile pour accéder au même serveur via un autre protocole (CIFS au lieu de MSSQLSvc)

RBCD — Resource-Based Constrained Delegation

  • RBCD renverse le modèle de contrôle de la délégation : au lieu que le compte qui délègue liste les services cibles, c'est la ressource cible qui liste les comptes autorisés à déléguer vers elle. Cela est configuré via l'attribut msDS-AllowedToActOnBehalfOfOtherIdentity sur l'objet Computer cible.
  • L'avantage sécuritaire : les admins de la ressource contrôlent qui peut se faire passer pour des utilisateurs pour y accéder — plus besoin de droits d'admin de domaine pour configurer ça.
  • Le vecteur d'attaque : si un attaquant a GenericWrite (ou GenericAll) sur un objet Computer dans AD, il peut modifier msDS-AllowedToActOnBehalfOfOtherIdentity pour y inscrire un compte machine qu'il contrôle. Ce compte contrôlé peut alors usurper Administrator pour accéder à la machine cible.
  • La création d'un compte machine est triviale : tout utilisateur de domaine peut en créer jusqu'à 10 par défaut (MachineAccountQuota = 10) — aucun droit admin requis.

RBCD — Attaque complète étape par étape

// Prérequis : GenericWrite sur WS01.corp.local

1. Créer un compte machine contrôlé (MachineAccountQuota = 10)

addcomputer.py corp.local/alice:Passw0rd -computer-name "FAKE$" -computer-pass "Azerty1!"

2. Récupérer le SID de FAKE$ et configurer RBCD sur WS01

rbcd.py corp.local/alice:Passw0rd -delegate-from "FAKE$" -delegate-to "WS01$" -action write

3. S4U2Self + S4U2Proxy : obtenir un ST admin pour WS01

getST.py corp.local/"FAKE$":Azerty1! -spn "cifs/ws01.corp.local" -impersonate Administrator

4. Utiliser le ST (KRB5CCNAME=Administrator.ccache)

secretsdump.py -k -no-pass ws01.corp.local → hashes locaux → mouvement latéral

RBCD — Via NTLM Relay (combinaison P4 + P6)

  • La combinaison NTLM Relay (Partie 4) + RBCD (Partie 6) est particulièrement puissante car elle ne nécessite aucun accès préalable à la machine cible — juste capturer l'authentification NTLM d'un compte ayant GenericWrite sur une machine.
  • ntlmrelayx en mode --delegate-access automatise entièrement les étapes 1 et 2 de l'attaque RBCD : il crée le compte machine et configure RBCD en une seule opération, le tout via le relay LDAP.
# Lancer ntlmrelayx en mode RBCD automatique
ntlmrelayx.py -t ldaps://dc01.corp.local \
  --delegate-access \
  --add-computer FAKE$ \
  --escalate-user attacker_user

# Une fois RBCD configuré, obtenir le ST et accéder à la cible
getST.py corp.local/"FAKE$":password -spn "cifs/ws01.corp.local" -impersonate Administrator
export KRB5CCNAME=Administrator@cifs_ws01.corp.local.ccache
secretsdump.py -k -no-pass ws01.corp.local
